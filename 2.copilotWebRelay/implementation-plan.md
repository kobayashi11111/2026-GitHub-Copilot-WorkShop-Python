# 段階的実装計画 (Implementation Plan)

ポモドーロタイマーアプリケーションの段階的な実装ロードマップです。各フェーズは独立して動作確認・テスト可能な単位になっています。

## Phase 0: 環境構築と最小構成 (1-2時間)

**目標**: プロジェクトの骨組みを作り、最小限の Flask アプリが動く状態にする。

**実装内容**:
- [ ] `1.pomodoro/` ディレクトリ構造の作成
- [ ] `requirements.txt` の作成（Flask, SQLAlchemy など）
- [ ] `app.py` にアプリファクトリ `create_app()` の実装
- [ ] 静的ファイル用のディレクトリ構造（`templates/`, `static/css/`, `static/js/`）
- [ ] 最小限の `index.html` テンプレート（"Hello Pomodoro" 表示）
- [ ] Flask 開発サーバーで動作確認

**確認方法**:
```bash
cd 1.pomodoro
python app.py
# ブラウザで http://localhost:5000 にアクセスして画面が表示される
```

---

## Phase 1: 基本タイマー UI（フロントエンドのみ）(3-4時間)

**目標**: サーバー連携なしで、ブラウザ上で動作するタイマーを完成させる。

**実装内容**:
- [ ] `static/js/timerEngine.js` の実装
  - `start()`, `pause()`, `reset()` メソッド
  - 時刻注入可能な設計（`clock` 引数）
  - 経過時間の正確な計算（ドリフト対策）
  - イベントコールバック (`onTick`, `onComplete`)
- [ ] `static/js/uiController.js` の実装
  - DOM 要素の取得と更新
  - ボタンイベントリスナー
  - タイマー表示の更新（MM:SS フォーマット）
- [ ] `static/css/style.css` の実装
  - モック画像に沿ったレイアウト
  - 円形プログレスバーの描画（SVG）
- [ ] `index.html` の完成
  - タイマー表示エリア
  - 開始/リセットボタン
  - 今日の進捗エリア（静的表示）

**確認方法**:
- タイマーが正確にカウントダウンする
- 開始/一時停止/リセットが動作する
- 円形プログレスバーがアニメーションする

**テスト**:
- [ ] `TimerEngine` の基本ユニットテスト作成（`vitest` または `jest`）

---

## Phase 2: バックエンド基盤と API (3-4時間)

**目標**: データベースと API エンドポイントを実装し、セッションを記録できるようにする。

**実装内容**:
- [ ] `models.py` の実装
  - `Session` モデル（id, start_time, end_time, planned_duration, actual_duration, status）
  - SQLAlchemy 初期化
- [ ] `api.py` の実装
  - `POST /api/session/start` エンドポイント
  - `POST /api/session/complete` エンドポイント
  - `POST /api/session/cancel` エンドポイント
  - `GET /api/stats/today` エンドポイント
- [ ] DB 初期化処理
  - マイグレーションスクリプト（または `db.create_all()`）
- [ ] `static/js/apiClient.js` の実装
  - fetch ベースの API 呼び出しラッパー
  - エラーハンドリング

**確認方法**:
```bash
# API テストツール（curl または Postman）で各エンドポイントを確認
curl -X POST http://localhost:5000/api/session/start \
  -H "Content-Type: application/json" \
  -d '{"planned_duration": 1500}'
```

**テスト**:
- [ ] `tests/conftest.py` 作成（pytest fixture: app, client, db）
- [ ] `tests/test_api.py` の基本テスト（各エンドポイントの正常系）

---

## Phase 3: フロント・バックエンド統合 (2-3時間)

**目標**: タイマーUIとバックエンドAPIを接続し、セッションが記録されるようにする。

**実装内容**:
- [ ] `uiController.js` に API 連携を追加
  - タイマー開始時に `/api/session/start` を呼ぶ
  - タイマー完了時に `/api/session/complete` を呼ぶ
  - キャンセル時に `/api/session/cancel` を呼ぶ
- [ ] 今日の進捗表示を動的に更新
  - ページロード時に `/api/stats/today` を取得
  - セッション完了後に再取得して更新
- [ ] エラーハンドリング
  - API 呼び出し失敗時のユーザーフィードバック

**確認方法**:
- タイマー完了後、リロードしても「今日の進捗」が保持されている
- ブラウザの開発者ツールで API リクエストを確認

**テスト**:
- [ ] E2E テスト（オプション）: Playwright で基本フロー確認

---

## Phase 4: ローカル永続化とオフライン対応 (2-3時間)

**目標**: ブラウザをリロードしてもタイマー状態を復元し、オフライン時も動作させる。

**実装内容**:
- [ ] `static/js/storage.js` の実装
  - `localStorage` への状態保存・読み込み
  - タイマー状態（開始時刻、経過時間）の保存
- [ ] `uiController.js` に状態復元処理を追加
  - ページロード時に `localStorage` から状態を読み込み
  - タイマー再開が可能なら自動再開
- [ ] オフラインキュー機能
  - API 送信失敗時に `localStorage` にキューイング
  - オンライン復帰時（`online` イベント）に再送

**確認方法**:
- タイマー動作中にリロードしても継続する
- ネットワークを切断してセッション完了→再接続後に記録される

---

## Phase 5: 通知・サウンド・設定 (2-3時間)

**目標**: ユーザー体験を向上させる機能を追加する。

**実装内容**:
- [ ] `static/js/notifier.js` の実装
  - ブラウザ通知 API の呼び出し
  - 通知権限のリクエスト
- [ ] サウンド再生
  - タイマー終了時のアラート音（`<audio>` タグ）
- [ ] 設定画面の実装
  - 作業時間、短休憩、長休憩の入力フォーム
  - `localStorage` への設定保存
  - 設定読み込みと適用

**確認方法**:
- タイマー終了時に通知とサウンドが鳴る
- 設定変更後、次回のタイマー起動時に反映される

## 推奨開発順序のまとめ

| Phase | 内容 | 所要時間（目安） | 優先度 |
|-------|------|-----------------|--------|
| 0 | 環境構築 | 1-2h | 必須 |
| 1 | フロント UI | 3-4h | 必須 |
| 2 | バックエンド API | 3-4h | 必須 |
| 3 | 統合 | 2-3h | 必須 |
| 4 | ローカル永続化 | 2-3h | 高 |
| 5 | 通知・設定 | 2-3h | 中 |
| 6 | テスト拡充 | 3-4h | 高 |
| 7 | CI/CD | 2-3h | 中 |
| 8 | 拡張機能 | 継続的 | 低 |

**合計**: 約20-30時間（コア機能のみ）
