# アーキテクチャ概要

## システム構成

ポモドーロタイマーアプリケーションは、以下の2つの主要モジュールで構成されています：

### 1. `app.py` - メインアプリケーション

**役割**: ユーザーインターフェースとタイマー機能の提供

**主要クラス**: `PomodoroTimer`

**責務**:
- コマンドラインインターフェースの提供
- ポモドーロタイマーのカウントダウン実行
- メニュー表示と操作フロー制御
- ゲーミフィケーション情報の表示

### 2. `gamification.py` - ゲーミフィケーション管理

**役割**: 経験値、レベル、バッジ、ストリーク、統計の管理

**主要クラス**: `GamificationManager`

**責務**:
- 経験値(XP)とレベルの計算・管理
- バッジの獲得判定と管理
- 連続日数(ストリーク)の追跡
- セッション履歴の記録
- 統計データの集計と提供
- データの永続化 (JSON形式)

## アーキテクチャパターン

### レイヤー構造

```
┌─────────────────────────────────┐
│    プレゼンテーション層          │
│    (app.py / PomodoroTimer)     │
│  - CLI インターフェース          │
│  - ユーザー入力処理              │
│  - 表示ロジック                  │
└─────────────────────────────────┘
              ↓
┌─────────────────────────────────┐
│      ビジネスロジック層          │
│  (gamification.py /              │
│   GamificationManager)           │
│  - XP/レベル計算                 │
│  - バッジ判定                    │
│  - ストリーク更新                │
│  - 統計集計                      │
└─────────────────────────────────┘
              ↓
┌─────────────────────────────────┐
│       データ永続化層             │
│    (pomodoro_data.json)          │
│  - JSON形式でのデータ保存        │
└─────────────────────────────────┘
```

### 責任の分離

- **app.py**: UI/UX に関する責務のみを担当
- **gamification.py**: ビジネスロジックとデータ管理を担当
- 両モジュール間は疎結合で、依存関係は一方向（app.py → gamification.py）

## データフロー

### ポモドーロ完了時のフロー

```
1. ユーザーがポモドーロを開始
   ↓
2. PomodoroTimer.start_pomodoro()
   ↓
3. countdown() でタイマー実行
   ↓
4. 完了後、GamificationManager.add_pomodoro() を呼び出し
   ↓
5. GamificationManager が以下を実行:
   - XP加算
   - レベルアップチェック
   - セッション記録
   - ストリーク更新
   - バッジチェック
   - データ保存
   ↓
6. 結果を PomodoroTimer に返却
   ↓
7. PomodoroTimer が結果を画面表示
```

## 技術仕様

### 言語とライブラリ

- **Python**: 3.x
- **標準ライブラリのみ使用**:
  - `json`: データ永続化
  - `os`: システム操作（画面クリア、ファイル存在確認）
  - `sys`: システム関連（現在は未使用）
  - `datetime`, `timedelta`: 日時処理
  - `time`: タイマー実行

### 外部依存関係

**実行時**: なし（標準ライブラリのみ）

**テスト時**:
- `pytest>=7.4.0`: テストフレームワーク
- `pytest-cov>=4.1.0`: カバレッジ測定
- `pytest-mock>=3.11.0`: モック機能

### データ永続化

- **形式**: JSON
- **ファイル名**: `pomodoro_data.json`（デフォルト）
- **エンコーディング**: UTF-8
- **保存タイミング**: ポモドーロ完了時に自動保存

## 設計の特徴

### 1. シンプルさ

- 外部ライブラリへの依存を最小化
- 標準ライブラリのみで完結
- 複雑なフレームワークを使用せず、シンプルなコマンドラインアプリとして実装

### 2. テスタビリティ

- クラスベースの設計でモックやスタブが容易
- 各メソッドが単一責任を持つ
- 包括的なユニットテストが実装済み（95%以上のカバレッジ）

### 3. 拡張性

- ゲーミフィケーション機能が独立したモジュール
- 新しいバッジタイプの追加が容易
- データ構造が柔軟（後方互換性を保ちながら拡張可能）

### 4. データ整合性

- ストリークの自動計算
- XPとレベルの自動連動
- バッジの重複獲得防止
- データファイル破損時のフォールバック処理

## エラーハンドリング

### データファイル読み込み時

- ファイルが存在しない → デフォルトデータで初期化
- JSONが破損している → デフォルトデータで初期化
- IOエラー → デフォルトデータで初期化

### ユーザー操作

- `Ctrl+C` でタイマー中断可能
- 無効なメニュー選択 → エラーメッセージ表示後、再入力

## パフォーマンス考慮事項

### メモリ使用

- セッション履歴はすべてメモリ上に保持
- 長期間使用すると履歴データが増大する可能性あり
- 現在の実装では履歴の削除・アーカイブ機能なし

### ファイルI/O

- ポモドーロ完了時のみファイル書き込み
- 読み込みは起動時のみ
- JSON形式のため、大量データでは読み書きに時間がかかる可能性

## セキュリティ

- **機密情報なし**: ユーザーの個人情報を保存しない
- **ローカルストレージのみ**: ネットワーク通信なし
- **ファイルアクセス**: 実行ディレクトリ内のみ

## 今後の拡張可能性

### 考えられる機能追加

1. **Webインターフェース**: Flask/FastAPIでWeb版を実装
2. **データベース化**: SQLiteやPostgreSQLを使用したデータ管理
3. **複数ユーザー対応**: ユーザー管理機能の追加
4. **データエクスポート**: CSV/PDF形式でのレポート出力
5. **通知機能**: デスクトップ通知やサウンド再生
6. **カスタマイズ**: ポモドーロ時間の柔軟な設定
7. **タスク管理**: ポモドーロとタスクの紐付け

### アーキテクチャの発展方向

現在の構造から以下のパターンへ発展可能：

```
現在: CLI + ビジネスロジック + JSONストレージ
  ↓
将来: (Web/CLI) + サービス層 + リポジトリ層 + DBストレージ
```

この場合、`GamificationManager` をサービス層として維持し、リポジトリパターンでデータアクセス層を分離することで、スムーズな移行が可能です。
